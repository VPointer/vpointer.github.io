Spring AOP笔记

AOP

面向切面编程(Aspect Oriented Programming, AOP)，基于OOP基础上的一种新的编程思想，指在程序运行期间，将『某段代码』动态地切入到『指定方法的指定位置』进行运行的编程方法。

这种写法的问题其实就是：核心功能(接口中的方法)和辅助功能(日志记录)高度耦合了，每次修改都要修改核心代码。

动态代理

有没有不动核心功能的代码，又能实现日志功能呢？其实是有的，JDK自带的就有解决办法，即动态代理，通过反射来实现上述功能，又不动核心代码。

但如果这种"原生"的动态代理真的牛叉的话，那哪还有Spring的事呢？这种"原生"的动态代理其实有如下非常严重的缺陷：

  1. 这玩意儿写着太难了；
  2. 最严重的缺陷：只有实现了接口的类才能使用动态代理，如果一个类并没有接口，那就用不了动态代理。因为动态代理之所以能关联到被代理对象，就是因为实现了相同的接口。

Spring框架的AOP

它的底层其实也是动态代理，但在Spring中，AOP的使用非常简单，没有接口的类也可以使用Spring AOP，一句创建代理的代码都不用写就可以实现AOP

Spring AOP的使用

前面的代码并没有切面类。仍以上述的那个动态代理类为例，其实将类中的那些输出语句提取出来就是一个切面类，这里将其命名为SysOutUtils。

要使用Spring AOP，需要涉及到了另外6个注解：

  • @Aspect:它用于标记切面类，也就是这里的LogUtils
  • @Before:前置通知
  • @After:后置通知
  • @AfterReturning:返回通知
  • @AfterThrowing:异常通知
  • @Around:环绕通知，Spring里面最强大的通知，上面4个通知的四合一，其实就相当于是自己写动态代理。
  • @PointCut:用于复用切入点表达式

Spring AOP的细节

  1. 测试中的bean的类型并不是MyCalculator类！而是Spring自动创建的一个动态代理类！这也说明，Spring底层用的依然是动态代理。这也是为什么测试代码中获取bean时要传入接口类，而不是本类。

    如果MyCalculator没有继承Calculator接口，那么将ioc.getBean()方法的参数改为MyCalculator.class，则上述代码依然可以运行！只是此时测试代码中bean的类型将是MyCalculator的一个内部类，这个内部类由cglib库创建，并实现了MaCalculator的全部方法。这也证明了，即使没有实现接口，Spring依然可以实现切面编程！

  2. 切入点表达式的写法：

    • *通配符：
      • 匹配一个或多个字符，可以用在切入点表达式的任意地方，包括参数列表中(方法重载的情况)，如果用在路径中，则表示匹配一层路径。例如execution(public int com.learning.*.MyCal*r.*(int, *))
      • *通配符不能用于修饰方法的权限！即上述的public等权限修饰符。如果想匹配任意权限，则不写权限既可(但即使这样也只能切public的，切不了private)；
    • ..：两个点，它有如下含义：
      • 表示匹配任意多个任意类型的参数。例如execution(public int com.learning.aspect.MyCalculator.*(..))
      • 表示匹配任意多层路径：例如execution(public int com.learning.*.MyCalculator.*(..))只表示匹配一层路径，而execution(public int com.learning..MyCalculator.*(..))则表示匹配任意层路径
    • 切记两种表达式：
      • 最精确的：execution(public int com.learning.aspect.MyCalculator.add(int, int))
      • 最糢糊的：execution(* *.*(..))或者execution(* *(..))，它们都表示任意包下的任意类的任意方法，千万别写这种！
    • &&, ||, !：切入点表达式还可以用逻辑运算符组合起来使用，含义和java中的含义一样,例如:execution(public int com.learning.*.MyCal*r.*(int, *))&&execution(*, *.*(..))

  3. 上述两个切面类中带有切面注解的方法就是通知方法。

  4. 通知方法的执行顺序：和我们的习惯有些出入：

    • 无异常情况的顺序：@Before —> @After —> @AfterReturning
    • 有异常情况的顺序：@Before —> @After —> @AfterThrowing

    别人底层就是这么写的，并不是我们用错了。

  5. 给切面方法设置JoinPoint形参可以用于获取连结点的信息，比如连结点的方法名和传入连结点的参数。

  6. 如果想要获取连接点的返回值，可以才注解中使用returning参数，并在通知方法的参数列表中加入一个形参用于接收返回值。不过，用于接收返回值的形参名必须和returning的参数值相同！

  7. 如果想要获取连接点抛出的异常，可以才注解中使用throwing参数，并在通知方法的参数列表中加入一个形参用于接收异常。不过，用于接收异常的形参名必须和throwing的参数值相同！

  8. Spring对通知方法的要求并不严格，即使通知方法有返回值，或者通知方法是private的都没有关系，依然可以正常运行。唯一有要求的就是通知方法的形参列表不能乱加参数，你加的每一个参数都要通过给注解传入相应的参数，告诉Spring，你添加的这个参数是用来干什么的。

  9. 注解中的切入点表达式可以单独抽去出来进行复用，例如上述的FileOutUtils类中传入切入点表达式的方式。具体方法就是在切入类中随便声明一个方法，例如上述的forAnno()方法，并在这个方法上使用@PointCut注解，@PointCut中的参数就是要复用的切入点表达式。当某个通知方法要复用这个切入点表达式时，传入这个带有@PointCut注解的方法名以及括号既可，例如@Before("forAnno()")

  10. 切面类执行顺序：如果有多个切面类，默认情况下，是按照切面类类名的ASCII码顺序来进入切面类的，但是可以通过@Order(数字)注解来控制切面的执行顺序，数字范围就是int的范围，数字越小，优先级越高。而离开切面类则是相反的顺序，也就是类似于栈，先进后出。切面类对于被代理的对象来说是一种包裹关系，优先级最高的切面类包裹在最外面，优先级最低的包裹在最里面，@Before注解表示"进入"，其他注解表示"出去"，那当然是最外面的先进入，最后才出去。从上述例子的输出也可以看到这一点。

最后还要重点介绍一下环绕通知@Around

环绕通知@Around

观察SysOutUtils.around()方法可以发现，环绕通知其实就相当于是自己写一个动态代理类，它完全接管了连结点的运行，也就是说，它能控制连结点究竟运不运行，要传入什么参数，要抛出什么异常，以及返回什么值。而要让连结点运行，我们还需要使用一个类型为ProceedingJoinPoint(JoinPoint的子类)的参数pjp，并调用它的.proceed()方法。

另一个点则是，环绕通知先于其他4个类型的通知执行。如果环绕通知存在，那么其他4个通知接收到的信息(比如参数，返回值，抛出的异常等)就都是环绕通知反馈给它们的。也就是说，原本add(1, 2)应该返回3，且不抛出异常，但在环绕通知中，我们可以让这个方法不执行；也可以让这个方法执行，但传入的参数由我们指定；还一个返回一个我们想让他返回的值；甚至可以让它抛出一些异常。

但需要提醒的是，只在同一个切面类中，环绕通知才优先于其他4个类型的通知执行，环绕通知只影响自己所在的切面类，影响不到其他切面类。例如上述结果中，SysOutUtils.around()方法总是先于同类中的其他通知方法执行，但FileOutUtils中的通知方法又“优先”于SysOutUtils中的所有通知方法。

这种级别的控制像不像我们亲自写动态代理？
