Maven学习笔记

Maven概述

一个对Maven比较正式的定义是：它是一个项目管理工具，它包含了一个项目对象模型(POM：Project Object Model)，一组标准集合，一个项目生命周期(Project Lifecycle)，一个依赖管理系统(Dependency Management System)，和用来运行定义在生命周期阶段中插件目标的逻辑。

通俗的讲，Maven解决的问题就是以下3点：

  • 提供了一套标准化的项目结构；
  • 提供了一套标准化的构建流程（编译，测试，打包，发布……）；
  • 提供了一套依赖管理机制。

依赖管理概述：

传统的Java项目，jar包都是包含在项目中的，这就使得往往代码其实很少，但最后导出的项目jar包却大了很多。而在Maven项目中，所依赖的jar包并不包含在最终的文件中，它包含的是jar包的“坐标”，当项目运行时，它会到仓库中找到所需的依赖。

Maven仓库涉及的几个概念：

  • 中央仓库repo1.maven.org：它包含了全球所有开源的jar包；
  • 本地仓库：当某个jar包首次被用到时，Maven会将该jar包从中央仓库下载到本地仓库。本地仓库的默认目录是用户目录/home下的.m2/repository；
  • 镜像仓库：众所周知的原因，国内访问Maven中央仓库有时候会很慢，此时可以设置Maven的镜像仓库。如何设置请看下一小节；
  • 私有仓库（私服）：公司内部如果不希望把源码和jar包放到公网上，那么可以搭建私有仓库。私有仓库总是在公司内部使用，它只需要在本地的~/.m2/settings.xml中配置好，使用方式和中央仓位没有任何区别。

解释：

  • YourProject：项目名，同时也是项目的根目录；
  • pom.xml：项目描述文件，非常重要的一个项目配置文件；
  • src/main/java：Java源码目录；
  • src/main/resource：资源文件目录；
  • src/main/webapp：页面资源，js，css等
  • src/test/java：测试文件的源码目录；
  • src/test/resource：测试资源目录；
  • target：存放所有编译、打包生成的文件。

所有的目录都是约定好的标准结构，切勿随意更改。

一个Maven工程的唯一标识符由groupId，artifactId和version构成：

  • groupId：类似于Java的包名，通常是公司或组织名；
  • artifactId：类似于Java的类名，通常是项目名；
  • version：版本号。以-SNAPSHOT结尾的版本号会被Maven视为开发版，开发版每次都会重复下载，这种SNAPSHOT版本只能用于内部私有的Maven repo，公开发布的版本不允许出现SNAPSHOT。

Maven通过对jar包进行PGP签名确保任何一个jar包一经发布就无法修改。修改已发布jar包的唯一方法是发布一个新版本。因此，某个jar包一旦被Maven下载过，即可永久地安全缓存在本地。

依赖关系

Maven定义了4种依赖关系，分别是compile、test、runtime和provided：

scope 说明 示例
compile 编译时需要用到该jar包（默认） commons-logging
test 编译Test时需要用到该jar包 junit
runtime 编译时不需要，但运行时需要用到 mysql
provided 编译时需要用到，但运行时由JDK或某个服务器提供 servlet-api

其中，默认的compile是最常用的，Maven会把这种类型的依赖直接放入classpath。

项目构建流程

Maven项目的生命周期(Lifecycle)由一系列阶段(Phase)构成，而一个phase又会触发一个或多个goal。

default生命周期

以内置的默认生命周期default为例，它包含以下phase：

  • validate
  • initialize
  • generate-sources
  • process-sources
  • generate-resources
  • process-resources
  • compile
  • process-classes
  • generate-test-sources
  • process-test-sources
  • generate-test-resources
  • process-test-resources
  • test-compile
  • process-test-classes
  • test
  • prepare-package
  • package
  • pre-integration-test
  • integration-test
  • post-integration-test
  • verify
  • install
  • deploy

如果执行mvn package命令，Maven就会执行default生命周期，它会从开始运行到package这个phase为止。其他的以此类推。

clean生命周期

这是Maven另一个常用的生命周期，它包含了3个phase：

  • pre-clean
  • clean （注意这个clean不是lifecycle而是phase）
  • post-clean

Goal

goal是Maven的最小人物单元，它的命名总是abc:xyz这种形式。执行一个phase又会触发一个或多个goal：

执行的Phase 对应执行的Goal
compile compiler:compile
test compiler:testCompile
surefile:test

常用构建命令

上述phase可以组合起来使用，以下是一些常用的命令：

  • mvn clean：清理所有生成的class和jar；
  • mvn clean compile：先清理，再执行到compile；
  • mvn clean test：先清理，再执行到test，因为执行test前必须执行compile，所以这里不必指定compile；
  • mvn clean package：先清理，再执行到package；
  • mvn install：会将生成的jar包或者war包安装到本地仓库；
  • mvn deploy：执行该命令前需要配置

使用插件

Maven本身并不知道该如何执行phase，实际上，对于每一个phase，Maven都是通过相应的插件(plugin)来执行的。例如，命令mvn compile，Maven是通过调用compiler这个自带的插件，然后通过它来执行默认的compiler:compile这个goal来完成编译的。

Maven内置了一些常用的标准插件，例如：

插件名称 对应执行的phase
clean clean
compiler compile
surefire test
jar package

Maven Wrapper

mvnw是Maven Wrapper的缩写。默认情况下，所有项目都会使用全局安装的Maven版本。但对于某些项目来说，它可能必须使用某个特定版本的Maven，这时候就可以使用Maven Wrapper，它将给这个特定的项目安装指定版本的Maven，而其他项目不受影响。类似于Python的虚拟环境。
